* Gemini answer for sites certificates
** Create a root-ca
*** Create private-key for root-ca
#+begin_src bash
openssl genpkey -algorithm RSA -pkeyopt rsa_keysize:4096 -out ca.key
#+end_src
*** Create Certificate for root-ca (auto-sign)
#+begin_src bash
openssl req -x509 -new -nodes -key ca.key -sha256 -days 3650 -out ca.crt
#+end_src
** Generate server certificate
*** Create private key and CSR
#+begin_src bash
openssl req -new -newkey rsa:2048 -nodes -keyout server.key -out server.csr
#+end_src
*** Configuration file
#+begin_src conf
authorityKeyIdentifier=keyid,issuer
basicConstraints=CA:FALSE
keyUsage=digitalSignature, nonRepudiation, keyEncipherment, dataEncipherment
subjectAltName=@alt_names

[alt_names]
IP.1 = 192.168.1.10  # VOTRE IP PRIVÉE ICI
#+end_src
*** Sign server certificate
#+begin_src bash
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial \
-out server.crt -days 730 -sha256 -extfile v3.ext
#+end_src
** Installation & browser trust
C'est une excellente précision. Si votre site est accessible uniquement via une adresse IP privée (par exemple, dans un réseau local ou un intranet, comme http://192.168.1.10), vous ne pouvez pas utiliser les autorités de certification publiques comme Let's Encrypt ou les CA commerciales, car elles exigent de pouvoir valider la propriété d'un nom de domaine public (FQDN).

La solution pour avoir du HTTPS dans ce contexte est de mettre en place votre propre Autorité de Certification Privée (PKI).

Voici les étapes à suivre pour auto-signer votre serveur sur une IP privée, en utilisant la même logique que nous avons abordée précédemment, mais en créant votre propre chaîne de confiance :

1. Créer une Autorité de Certification Privée (Root CA) 🔒
Vous devez d'abord devenir votre propre autorité de certification. Vous allez générer une paire de clés et un certificat CA qui vous permettront de signer d'autres certificats par la suite.

Générer la Clé Privée de la CA :

Bash

openssl genpkey -algorithm RSA -pkeyopt rsa_keysize:4096 -out ca.key
Créer le Certificat Root CA (Auto-signé) :

Bash

openssl req -x509 -new -nodes -key ca.key -sha256 -days 3650 -out ca.crt
ca.key est la clé privée de votre CA (gardez-la très sécurisée).

ca.crt est le certificat public de votre CA.

2. Générer le Certificat du Serveur Web 💻
C'est l'étape où vous créez le certificat pour l'adresse IP privée spécifique (ex: 192.168.1.10).

A. Créer la Clé Privée et le CSR du Serveur
Vous générez une nouvelle clé privée pour votre serveur web et une demande de signature (CSR).

Bash

openssl req -new -newkey rsa:2048 -nodes -keyout server.key -out server.csr
B. Le Fichier de Configuration (Crucial pour l'IP) 📄
Pour que le navigateur accepte l'IP, vous devez inclure l'adresse dans l'extension Subject Alternative Name (SAN). Vous créez un fichier de configuration temporaire (ex: v3.ext) :

Ini, TOML

authorityKeyIdentifier=keyid,issuer
basicConstraints=CA:FALSE
keyUsage=digitalSignature, nonRepudiation, keyEncipherment, dataEncipherment
subjectAltName=@alt_names

[alt_names]
IP.1 = 192.168.1.10  # VOTRE IP PRIVÉE ICI
C. Signature du Certificat du Serveur
Vous utilisez votre CA privée (ca.crt et ca.key) pour signer la demande de votre serveur (server.csr) en utilisant le fichier d'extension SAN (v3.ext).

Bash

openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial \
-out server.crt -days 730 -sha256 -extfile v3.ext
server.crt est votre certificat HTTPS final, signé par votre propre CA.

3. Installation et Confiance du Navigateur 🤝
L'installation sur le serveur web se fait comme pour un certificat public (voir l'étape 3 de la réponse précédente : configurer Apache/Nginx avec server.key et server.crt).

Cependant, comme le certificat n'est pas signé par une CA reconnue mondialement, les navigateurs vont afficher un avertissement de sécurité.

Pour que l'avertissement disparaisse :

Distribuez le fichier ca.crt (le certificat de l'Autorité Racine privée que vous avez créé).

Installez ce fichier ca.crt dans le magasin de certificats "Autorités Racines de Confiance" (Trusted Root Authorities) de chaque machine cliente (votre ordinateur, celui de vos collègues, etc.) qui doit accéder au site.

En installant votre propre CA dans les magasins de confiance des clients, vous leur dites : "Faites confiance à tout certificat signé par cette CA." Votre navigateur reconnaîtra alors votre server.crt comme valide.
